# /post-review — ブログ記事監査スキル

## 概要

ブログ記事を `published: true` にする前の最終チェックとして機能する。
`/code-review` がコードを審査するように、記事を5軸で審査して重要度付きの指摘一覧を出す。

---

## 事前チェック

```bash
git status
```

- Gitリポジトリでなければ「Gitリポジトリが見当たりません」と伝えて終了。

```bash
git log main..HEAD --oneline
```

- コミットがなければ「コミットが見当たりません。先にコミットを作成してください」と伝えて終了。

## Step 1: 対象記事の特定

`$ARGUMENTS` を確認する。

- **指定あり**: `posts/<slug>.md` もしくは絶対パスとして解決してファイルを読む
- **指定なし**:変更があったファイルに対して、1つずつ検証を実行

---

## Step 2: Frontmatter 検証

以下のルールに照らし合わせて1件ずつ `✅` / `❌` を付ける。

| 項目               | ルール                                                         |
| ------------------ | -------------------------------------------------------------- |
| `title`            | 存在・文字列・空でない                                         |
| `category`         | `tech` / `life` / `idea` のいずれか                            |
| `createdAt`        | `YYYY-MM-DD` 形式かつ有効な日付                                |
| `updatedAt`        | 存在する場合は `YYYY-MM-DD` 形式かつ有効な日付                 |
| `published`        | boolean 型                                                     |
| slug（ファイル名） | kebab-case（英小文字・数字・ハイフンのみ、アンダースコア不可） |

**Frontmatter に1つでも ❌ があれば、そこで High 指摘を記録する。**

---

## Step 3: 5軸レビュー

記事本文を以下の5軸で読み、指摘事項を収集する。

### 軸1: 技術的正確性

- コードブロック内のコマンド・コードが文脈と矛盾していないか
- ライブラリ名・関数名・API名が誤っていないか
- インストールコマンドのパッケージ名や手順の順序が正しいか
- バージョン番号の指定があれば明らかな誤りがないか

### 軸2: 構成・読みやすさ

- 見出しレベルのルール: `#`（h1）は**使用禁止**（タイトルは frontmatter で指定）。`##` → `###` の順で使う
- 導入部: 記事の目的・対象読者・得られること が最初のセクションで明確か
- 結論・まとめ: 記事の締めが存在し、読者に次のアクションや要点を伝えているか
- 段落の流れ: 前後の説明が飛躍なくつながっているか

### 軸3: Markdown 品質

- コードブロックに言語指定があるか（例: ` ```ts `, ` ```bash ` etc.）
- 画像パスは `/画像名` 形式か（`public/` ディレクトリ参照のため先頭スラッシュ必須）
- 内部リンクは `/tech/<slug>` または `/<category>/<slug>` 形式か（`http://localhost` や相対パスは不可）
- リンクの `[text](url)` 記法が崩れていないか

### 軸4: タイトル・カテゴリ適切性

- タイトルが記事の内容を的確・具体的に表しているか（曖昧すぎ・広すぎは指摘）
- `category` の選択が内容に合っているか（技術解説 → `tech`、思想・考察 → `idea`、趣味・生活 → `life`）

### 軸5: 一貫性・重複

- 同じ説明・手順が複数箇所で繰り返されていないか
- 前の節と矛盾する記述（例: 「使わない」と言っていたものを後で使っているなど）がないか

---

## Step 4: 重要度スコアリング

各指摘に以下のいずれかを付与する。

| 重要度    | 基準                                                                                 |
| --------- | ------------------------------------------------------------------------------------ |
| 🔴 High   | 公開に影響する問題。Frontmatter エラー・技術的誤り・壊れたリンクなど                 |
| 🟡 Medium | 読者体験に影響するが公開は可。構成の不備・不明瞭な導入・コードブロック無言語指定など |
| 🟢 Low    | ニュアンス・好みレベルの改善提案                                                     |

---

## Step 5: レポート出力

変更があったファイルごとに以下のフォーマットで出力する。

```
## /post-review: <title>

### Frontmatter チェック
- ✅ title: "..."
- ✅ category: tech
- ❌ createdAt: "2026/06/09" → YYYY-MM-DD 形式にする必要あり
- ...

### 指摘事項（<件数> 件）

| 重要度 | 軸 | 場所 | 指摘内容 | 修正案 |
|---|---|---|---|---|
| 🔴 High | Frontmatter | createdAt | ... | ... |
| 🟡 Medium | 構成 | 導入部 | ... | ... |
| ...

### 総評
**公開可** ✅  ← High が 0 件の場合
または
**要修正** ❌ （High: N 件、Medium: M 件、Low: K 件）← High が 1 件以上の場合


```

High が 0 件であれば「公開可」、1 件以上であれば「要修正」と判定する。

---

## 禁止事項

- 指摘が0件のときも「問題なし」とだけ言って終わらない。全軸について「確認した・問題なし」を明示すること
- 記事の修正は行わない。フィードバックのみ提供する
- 曖昧な指摘（「改善の余地あり」など）を出さない。具体的な箇所と修正案をセットで示す
- ポジティブな感想を必ずいれる
