# devnote

会話・セッション・PR Reviewまたは `/think` セッションの議論を `~/dev/LIFE/notes/` に保存する。

## 引数

```
$ARGUMENTS
```

- **引数なし**: 会話全体を対象にメモを作成する
- **引数あり** (例: `エラー調査の部分だけ`, `DB設計の議論のみ`): その指示に従い会話の該当部分だけを対象にしてメモを作成する

## 手順

### Step 1: リポジトリの存在確認

最初に `~/dev/LIFE` が存在するか確認する：

```bash
test -d ~/dev/LIFE
```

存在しない場合は**即座に中断**し、以下のメッセージを表示する：

```
エラー: ~/dev/LIFE が見つかりません。
先に ~/dev/LIFE リポジトリをクローンまたは作成してから実行してください。
```

存在する場合のみ Step 2 以降に進む。

### Step 2: メタ情報の決定

以下を決める：

**対象範囲とタイトル**

「会話」とは、ユーザーとアシスタントの**実際のチャットのやり取りのみ**を指す。システムコンテキスト・git status・`<system-reminder>` タグの内容・ファイルの内容などは「会話」に含めない。

- `$ARGUMENTS` がある場合: その指示が示す会話の該当部分のみを対象とし、その内容からタイトルを自動生成する
- `$ARGUMENTS` がない場合: 会話全体を対象とし、主題を端的に表す**日本語**タイトルを生成する（例: `React useEffectの無限ループを修正`）
- タイトルは**会話の中で議論されたテーマ**から生成する。コードの作業状態や git 履歴から推測してタイトルを付けてはならない。

**タイプの自動判定**（以下の優先順位で判断）

| タイプ | 判定条件 |
|--------|----------|
| `think` | `/think` コマンドによるセッションが会話中に存在する（最優先） |
| `debug` | エラーメッセージ・スタックトレース・バグ再現手順が含まれる |
| `solution` | 問題が解決済みで、解決策が明確に存在する |
| `question` | 疑問・調査中で、まだ答えが出ていない |
| `pr-review` | PullRequestのレビュー内容 |
| `note` | 上記に当てはまらない学習メモ・調査結果など |

**タグの自動生成**
- 会話の内容から関連するキーワードを3〜5個抽出する
- 英小文字・ハイフン区切りで統一する（例: `react`, `use-effect`, `hooks`）

### Step 3: センシティブ情報のスクリーニング

まとめを生成する前に、会話中に含まれうる以下の情報を**一切含めない**ようにする：

- API キー・トークン・シークレット（例: `sk-...`、`ghp_...`、Bearer トークンなど）
- 環境変数の値（`.env` ファイルの内容、`export FOO=bar` の値部分）
- パスワード・認証情報
- 上記を示唆するキーと値のペア

変数名や「APIキーを使う」という設計上の言及は残してよい。値だけを除外する。

### Step 4: ファイルパスの決定

- **保存先:** `~/dev/LIFE/notes/`
- **日時プレフィックスの取得:**

```bash
date +%Y%m%d%H%M
```

- **ファイル名:** `{YYYYMMDDHHMI}-{slug}.md`
  - タイトルから記号・スペースを除いた**日本語のまま**の `slug` を付ける
  - 例: タイトルが `React useEffectの無限ループを修正`、日時が `202605141430` → `202605141430-ReactuseEffectの無限ループを修正.md`
  - 英数字はそのまま残し、スペース・句読点・記号は除去する

`~/dev/LIFE/notes/` が存在しない場合は作成する：

```bash
mkdir -p ~/dev/LIFE/zettelkasten/literature
```

### Step 5: マークダウンの生成

タイプに応じてテンプレートを切り替える。
**各セクションは会話に情報があるものだけ含める（空セクションは省略）。**

#### `think` タイプの場合

````markdown
---
title: {タイトル}
date: {YYYY-MM-DD}
type: think
tags: [{tag1}, {tag2}, ...]
---

# {タイトル}

## 背景・経緯

{なぜこの議論が始まったか}

## 会話の流れ

{大まかな会話の流れ,ユーザーがどのような意見を出し、Claudeがどの用に答えたか}

## 出た案・選択肢

{議論の中で挙がったアイデアや選択肢を箇条書き}

## 反論・懸念点

{指摘された問題点やトレードオフ}

## 決定・方向性

{合意した内容、採用した方針}


````

#### `think` 以外のタイプの場合

````markdown
---
title: {タイトル}
date: {YYYY-MM-DD}
type: {debug | solution | question | note}
tags: [{tag1}, {tag2}, ...]
---

# {タイトル}

## 背景・経緯

{何が起きていたか、何を知りたかったか}

## 会話の流れ

{大まかな会話の流れ,ユーザーがどのような意見を出し、Claudeがどの用に答えたか}

## 環境

```
{OS / 言語 / フレームワーク / バージョン情報など}
```

## エラー・症状

```
{エラーメッセージ、スタックトレース、再現手順など}
```

（debugタイプのみ。それ以外は省略）

## 調査・試したこと

{試したアプローチ、ハマりポイント、調査過程}

（箇条書きや番号リストを適宜使う）

## 解決策・結論

{最終的にどう解決したか、または何が分かったか}

```{言語}
{解決策のコード、設定、コマンドなど}
```

## 参考

- {関連リンク、ドキュメント、参照した記事}
````

### Step 6: ファイルの書き込み

Write ツールでファイルを作成する。

### Step 7: Git操作（commit・push）

`~/dev/LIFE` に移動して以下の手順で git 操作を行う。ブランチは作成せず、**main に直接コミットする**。

**手順**

```bash
cd ~/dev/LIFE

# 1. mainブランチに切り替え
git checkout main

# 2. ファイルをステージング
git add notes/{YYYYMMDDHHMI}-{slug}.md

# 3. コミット
git commit -m "note: {タイトル}"

# 4. プッシュ
git push origin main
```

- push に失敗した場合はエラーメッセージを表示し、それ以降の処理は中断する

### Step 8: 完了報告

以下の形式で報告する：

```
保存しました: ~/dev/LIFE/notes/{YYYYMMDDHHMI}-{slug}.md
タイプ: {type} | タグ: {tags}
main → push済み
```
