---

DESIGN.mdは、ブランド・デザインシステムをAIエージェントに伝えるための自己完結型フォーマット。YAML frontmatter（機械可読トークン）とMarkdown本文（人間可読の説明）の2部構成で、詳細な仕様は `references/spec.md` を参照すること（トークンスキーマ、9セクションの定義・順序、記述例を網羅している）。**生成作業に入る前に必ず一度 `references/spec.md` を読み込むこと。**

以下の4フェーズで進める。各フェーズを飛ばさず順番に実行すること。

## フェーズ1: 情報収集

デザイントークンの手がかりを、使える範囲で以下すべてから集める。

1. **既存コードベース解析**: カレントディレクトリ以下に既存プロジェクトがあれば、CSS変数（`:root { --* }`）、Tailwind設定（`tailwind.config.*`のtheme）、デザイントークンファイル（`tokens.json`等）、主要コンポーネント実装（ボタン・入力欄・カード等）を検索し、実際に使われている色・フォント・角丸・スペーシング値を抽出する。
2. **参考UI/サービスの調査**: ユーザーに「参考にしたいUI・サービスはあるか」を尋ねる。
   - 具体的なURLが提示された場合 → WebFetchでそのページのHTML/CSSや見た目の特徴（配色・フォント・レイアウト・角丸・余白・コンポーネントの雰囲気）を解析する。
   - サービス名など固有名のみの場合 → WebSearchでそのサービスの公式サイトURLやブランドガイドライン記事を特定し、見つかったURLをWebFetchで解析する。
   - WebFetchが使えない、または解析が困難な場合は、検索結果の説明文やブログ記事の記述からトークンの見当をつける。
3. **対話ヒアリング**: 上記だけでは埋まらない定性的な情報（ブランドパーソナリティ、対象ユーザー、UIが与えたい感情、参考にした要素のうち特に重視する点など）をAskUserQuestionで確認する。1問ずつ、推奨案を添えて聞く。

情報源が競合する場合（コードベースの実装値とユーザーが指定した参考UIの雰囲気が食い違う場合など）は、ユーザーにどちらを優先するか確認する。

## フェーズ2: ドラフト提示

収集した内容を、`colors` / `typography` / `rounded` / `spacing` / `components`ごとに表形式でまとめ、ユーザーに提示する。この段階ではYAML全体は書かず、値と簡単な用途説明（例: `primary: #1A1C1E` — 見出し・本文用の深いインク色）を並べる程度でよい。

ユーザーからの修正指示を反映し、承認が得られるまで繰り返す。承認が得られたら次フェーズへ進む。

## フェーズ3: 生成

`references/spec.md` の仕様に厳密に従い、以下を生成する。

- **YAML frontmatter**: `version: alpha`, `name`, `description`, `colors`（`primary`は必須）, `typography`, `rounded`, `spacing`, `components` を、承認されたドラフトに基づき記述。トークン名は英語（`primary`, `body-md`等）。可能な箇所は `{colors.primary}` のようなトークン参照を使い重複を避ける。
- **Markdown本文**: 9セクション（Overview / Colors / Typography / Layout / Elevation & Depth / Shapes / Components / Do's and Don'ts）を規定順序で`##`見出しにより記述。本文の説明文は日本語で書く（トークン名自体は英語のまま）。関連情報がなければそのセクションは省略してよい。
  - **Components**: フェーズ1で実際に検出できたコンポーネントを優先して記載する。検出が乏しい場合はbutton/input/card/chip/modal等の一般的なコンポーネントセットで補い、その旨が分かるようにする。
  - **Do's and Don'ts**: 定めた主要な色の組み合わせ（例: 本文テキスト色×背景色）についてコントラスト比を計算し、WCAG AA基準（通常テキスト4.5:1、大きいテキスト3:1）を満たすか確認する。満たさない組み合わせがあれば「Don't」項目として明記し、代替案を提案する。

## フェーズ4: 保存

カレント作業ディレクトリ直下に `DESIGN.md` が既に存在するか確認する。

- **存在する場合**: 既存内容と新規生成内容の差分を提示し、上書きしてよいかユーザーに確認してから書き込む。
- **存在しない場合**: そのまま新規作成する。

保存後、生成した `DESIGN.md` のパスと概要（主要トークンの要約）をユーザーに報告する。
