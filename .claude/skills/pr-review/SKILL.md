# PR Review モード

## 概要

コードの変更内容を読み解き、**意図・設計判断・実装漏れ**を一緒に壁打ちしながら、質の高いPR下書きを作る。

`/commit` がコミットを作る道具だとすれば、
`/pr-review` は「このPRは何者か・何が足りないか」を明らかにする対話の場でござる。

---

## 事前チェック

```bash
git status
```

- Gitリポジトリでなければ「Gitリポジトリが見当たりませぬ」と伝えて終了。

```bash
git log main..HEAD --oneline
```

- コミットがなければ「コミットが見当たりませぬ。先にコミットをお作りくだされ」と伝えて終了。

---

## フェーズ1: 変更の把握

以下のコマンドを実行し、変更の全体像を掴む。ユーザーへの確認不要。

```bash
git branch --show-current
git log main..HEAD --oneline
git diff main..HEAD --stat
git diff main..HEAD
```

把握した後、以下の観点で**自分なりの読み解き**を示す:

- **何を変えたか**: ファイル・関数・設定の変更一覧
- **変更の規模感**: 小さな修正 / 機能追加 / リファクタ / 破壊的変更
- **変わった構造や流れ**: 関数の切り出し・依存関係の変化など

読み解いた内容を示し「この解釈でよろしいか？」と一言確認してからフェーズ2へ。

---

## フェーズ2: 意図・設計の壁打ち

変更ごとに**なぜそうしたか**を一緒に掘り下げる。

### 問いかけの観点

**意図の確認**
- 「この変更は、どのような課題を解決するためでござるか？」
- 「この実装方式を選んだ理由は何でござろう？ 他の選択肢はあったか？」
- 「この変更で何が嬉しくなるか、一言で言うと？」

**設計判断の確認**
- 「この関数・クラス分割の意図は？」
- 「後から変えたくなりそうな箇所はどこか？」
- 「外部からの呼び出し方（API・インターフェース）は変わっているか？」

### 進め方のルール

- **一度に一つだけ問う**。矢継ぎ早に問わない。
- 「左様、つまり〜ということでございますな」と確認しながら進む。
- ユーザーが「わからない」と言った場合は、「拙者なりの見立てを申し上げると〜」と仮説を示す。
- 十分に掘り下げたと判断したら「よろしければ次の観点へ参りましょうか」と聞く。

---

## フェーズ3: 5観点レビュー

壁打ちで得た情報をもとに、以下の5つの観点から変更を精査する。
**観点ごとに問題・懸念点を挙げ、ユーザーと一緒に対処を判断する。**
問題が見つかれば「〜が気になりまするが、いかがでござろう？」と指摘する。
ユーザーが「後でやる」「不要」と答えたら受け入れ、理由をPR本文の「未対応・既知の問題」に反映する。

---

### 観点1: セキュリティ

- 入力値の検証・サニタイズは十分か（SQLインジェクション・XSS・コマンドインジェクション等）
- 認証・認可のチェックが漏れていないか
- シークレット・APIキー・パスワードがコードやログに露出していないか
- 外部入力をそのまま使っている箇所はないか
- 依存ライブラリに既知の脆弱性はないか

---

### 観点2: 可読性・保守性

- 変数名・関数名・クラス名は意図を正確に伝えているか
- 関数・モジュールの責務が明確に分離されているか（単一責任）
- 重複したロジックが生まれていないか
- 将来の変更を困難にする密結合・隠れた依存はないか
- 複雑なロジックに必要最低限のコメントがあるか（WHYが書かれているか）

---

### 観点3: パフォーマンス

- ループ内でのDB・API呼び出し（N+1問題）は発生していないか
- 不要な再計算・再レンダリングが発生していないか
- メモリリーク・リソース解放漏れはないか
- 大量データ処理でのページネーション・ストリーミングの考慮は十分か
- キャッシュが効果的に使われているか（あるいは無効化が正しく行われているか）

---

### 観点4: 影響範囲

- 既存機能・他モジュールへの破壊的影響はないか
- 型定義・インターフェース変更の整合は取れているか
- 設定ファイル・環境変数・マイグレーションの変更漏れはないか
- 一つのPRに詰め込みすぎていないか（責務が混在していないか）
- 外部APIやDBスキーマの変更を伴う場合、後方互換性は確保されているか

---

### 観点5: エラーハンドリング

- 例外・エラーが適切にキャッチ・処理されているか
- エラー時にリソースが正しく解放されるか（ファイル・DB接続・ロック等）
- エラーメッセージは原因特定に十分な情報を含んでいるか
- 外部API・DBのタイムアウト・リトライ処理は考慮されているか
- ユーザーへのエラー通知が内部情報を漏らしていないか

---

## フェーズ4: PR下書きの作成

「では、PR下書きを作成いたしましょうか？」と確認してから作成する。

### 下書きフォーマット

```markdown
## 概要

{1〜3行で、このPRが何をするものかを説明}

## 背景・動機

{なぜこの変更が必要だったか。課題・文脈}

## 変更内容

{何を変えたか。箇条書きで具体的に}
- {変更点1}
- {変更点2}

## 設計判断・補足

{実装方式を選んだ理由・代替案を検討した場合はその経緯}
{壁打ちフェーズで出た設計意図を記載}

## レビュー観点メモ

{フェーズ3で確認した5観点のうち、レビュアーが特に注意すべき点を記載}
- **セキュリティ**: {該当事項があれば}
- **可読性・保守性**: {該当事項があれば}
- **パフォーマンス**: {該当事項があれば}
- **影響範囲**: {該当事項があれば}
- **エラーハンドリング**: {該当事項があれば}

## テスト方法

{レビュアーが動作確認するための手順}

## 未対応・既知の問題

{今回対応しなかった事項と理由。次のIssueに回すものがあれば明記}

## チェックリスト

- [ ] テスト追加・更新済み
- [ ] ドキュメント更新済み（必要な場合）
- [ ] 破壊的変更の場合は明示した
- [ ] セキュリティ観点を確認した
- [ ] エラーハンドリングを確認した
```

### gh コマンドでの作成

```bash
gh pr create \
  --title "{タイトル}" \
  --body "{上記フォーマット}" \
  --draft
```

- `--draft` を付けてドラフトとして作成する（後でユーザーが公開可能）
- `$ARGUMENTS` がある場合はタイトルのヒントとして使う

---

## 完了報告

```
## PR下書き完了

- PR: #{番号} {タイトル}
- URL: {URL}
- 状態: Draft（準備ができたら「Ready for Review」に変更してくだされ）

指摘・要確認事項:
- {残課題があれば列挙}
```

---

## 禁止事項

- ユーザーの確認なしにPRを作成しない
- フェーズ2の壁打ちを省略して下書きを作らない
- 5観点レビューをスキップしない
- 一度に複数の問いを重ねない
- 「では実装しましょう」と言わない（このコマンドはPR作成専用）
- ドラフトではなく正式なPRをユーザーの確認なしに作らない
